# Releasing Calibration Scope

Releases are cut by [dist](https://github.com/axodotdev/cargo-dist) (v0.32.0,
pinned in [dist-workspace.toml](../dist-workspace.toml)). Pushing a version
tag runs [.github/workflows/release.yml](../.github/workflows/release.yml)
(generated by `dist generate` — regenerate, never hand-edit) and produces, on
the GitHub Release:

- `calibration-scope-{aarch64,x86_64}-apple-darwin.tar.xz` and
  `calibration-scope-x86_64-unknown-linux-gnu.tar.xz` — each carrying the
  `calibration-scope-dashboard` binary (assets embedded — see
  `src/embedded.rs` — so the binary alone is a working instrument)
- `calibration-scope-installer.sh` — the generated shell installer
- `calibration-scope.rb` — the Homebrew formula, auto-pushed to the tap
  (on beta tags too: `publish-prereleases = true`)
- `sha256.sum` + per-artifact `.sha256` checksums, `source.tar.gz`, and
  GitHub build-provenance attestations (verify with `gh attestation verify`)

Versions with a prerelease suffix (`v0.1.0-beta.1`) are automatically marked
**prerelease** on GitHub — that is the released form of the site's BETA tag.

## Two rules that protect the evidence chain

1. **Distributed binaries come ONLY from the CI release workflow.** rust-embed
   embeds from *disk*, not from git — a locally built binary bakes in whatever
   untracked files happen to sit under `assets/` at build time. CI builds from
   a clean checkout, so only tracked files ship.
2. **`assets/` is publish-on-commit.** Everything tracked under `assets/` at
   tag time is permanently embedded inside public artifacts; deleting a file
   from disk later does not remove it from released binaries.

## One-time setup (before the first tag)

1. **Create the tap repo**: an empty public repo named exactly
   `IT-Help-San-Diego/homebrew-tap` (the `homebrew-` prefix is what makes
   `brew install it-help-san-diego/tap/calibration-scope` resolve).
2. **Mint the tap token**: a fine-grained GitHub PAT with **Contents:
   read/write on the tap repo only**, saved as the `HOMEBREW_TAP_TOKEN`
   secret on *this* repo (Settings → Secrets and variables → Actions). The
   release workflow uses it to push the updated formula on every release.

Skipping these does not block the release: the GitHub Release, binaries,
checksums, and shell installer all publish first; only the
`publish-homebrew-formula` job then fails (and the announce step is skipped).

## Cutting a release

First set the new version in `Cargo.toml` `[package] version` and refresh the
lockfile with `cargo check --release` — check, NOT `cargo build --release`: a
release build here would overwrite `target/release/calibration-scope-dashboard`,
the exact binary the launchd service (`ai.hermes.calibration-scope-dashboard`)
loads on its next restart.

Then the closer (one line, closes on its own — Carey's one-liner rule: no
editors, no follow-up commands, `&&` all the way so any failure stops the
chain; `gh run watch --exit-status` blocks until CI is green and kills the
chain if it isn't, which is what makes tagging in the same breath safe):

```bash
git commit -am "release: vX.Y.Z-beta.N" && git push && sleep 5 && gh run watch --exit-status $(gh run list --branch main --limit 1 --json databaseId --jq '.[0].databaseId') && git tag vX.Y.Z-beta.N && git push origin vX.Y.Z-beta.N
```

The tag must match `Cargo.toml`'s version exactly (dist errors out if not —
that mismatch check is a feature). The CI-watch link matters because tag
pushes do NOT run the main CI gates (`ci.yml` triggers on branch pushes and
PRs only) — the tag must land on an already-green main commit, and the chain
enforces that instead of trusting a human to wait.

## Install lines this produces

```bash
# Shell installer, pinned to a specific release (macOS + Linux).
# NOTE: releases/latest/download/... resolves only once a NON-prerelease
# exists — while every release is a beta, `latest` 404s, so pin the tag:
curl -LsSf https://github.com/IT-Help-San-Diego/calibration-scope/releases/download/v0.1.0-beta.1/calibration-scope-installer.sh | sh

# Homebrew (installs postgresql@17 alongside, per dist-workspace.toml)
brew install it-help-san-diego/tap/calibration-scope
```

Both install the self-contained binary only. The runtime prerequisites stay
what they are: a reachable PostgreSQL (`DATABASE_URL`) and LM Studio for
local-model runs. The planned `calibrationscope.com/install.sh` wrapper (see
site Get Started section) will orchestrate those on top of these artifacts —
and can always point at the newest release (including betas) via the GitHub
API rather than `releases/latest`.

Platform floor: the Linux binary is built on ubuntu-22.04, so it needs
glibc ≥ 2.35. If older distros ever matter, add a musl target or a build
container in `dist-workspace.toml`.

## Changing release config

Edit `dist-workspace.toml`, then run `dist generate` and commit both it and
the regenerated `release.yml` together. `dist plan` previews what a tag would
produce without releasing anything.
