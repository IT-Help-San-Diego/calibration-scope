🔬 TEST PROTOCOL — read before running ▾
Environment controls
  • Open LM Studio → Developer (local server) tab and watch models load and eject in real time. Observe — don't touch. Loading or ejecting models by hand mid-run invalidates the measurement.
  • Close heavyweight apps before a run. Every GB another app holds is a GB the model under test can't use, and background CPU spikes contaminate latency numbers.
  • Testing your working setup instead? That's a valid experiment — just keep the same apps open across every model so runs stay comparable. Consistency is the control.
How the measurement works
  • Clean room: before each local run, every loaded model is ejected, only the target is loaded, and residency in RAM is verified by polling — never assumed.
  • No answer leakage: the expected answer lives only in the database. The model is never shown the ground truth it's scored against.
  • Objective scoring: verdicts come from comparing the model's actual output to ground truth (exact text, spatial relations, tool-call shape) — never from asking a model its opinion.
  • N=3 trials per test: one pass can be luck. Capability axes (Vision / Tools / Reasoning) score PASS / FAIL / FLAKY — can it do the job? Security scores SAFE / UNSAFE / FLAKY — did it resist? Different questions, different words.
  • Speed is a measurement, not a footnote: every axis shows average response time. A model that's right but takes 30 seconds is a different answer than one that's right in 2 — that's the ChatGPT-feel gap, measured honestly.
  • Evidence, not claims: every trial stores the exact prompt, raw response, and latency, sealed with a SHA-3 hash. You can audit every verdict.
Why this exists
  • Standardized benchmarks (MMLU & friends) are public — models have seen them, so cheating is possible. We still watch those numbers, but we don't trust them alone.
  • The cross-check is your own human test: build one in the Test Builder with your own image and question the model has never seen. A good test has one objective, verifiable answer that isn't contained in the question.
  • Local AI on your own hardware is real. The people saying it can't be done are missing method, not hardware. This page is the method — every number on it is one you can reproduce yourself.

🧭 Hermes Agent Cloud/Local Setup — read this before you benchmark
This dashboard talks to LM Studio directly, over the network — never through Hermes Desktop. That's why the science stays clean. Here's exactly how to configure Hermes so cloud and local both work correctly.
0. Accept reality first — measured on this machine, right now

Good science starts with gathering intelligence about what you actually have — before configuring anything. Every number below was measured from this machine when you opened this page (hover any value for the exact command it came from — every number has a receipt). Nothing here is a spec-sheet claim.

Measuring…

The mental model behind the budget line: in 2026, ~16–32GB is "the system plus an app or two." Around 64GB is where normal computing — the OS plus everything you leave open — actually lives comfortably. Whatever your machine has above that living space is what's honestly available to an AI stack, capped by macOS' GPU-wired memory ceiling. That's a heuristic, clearly labeled — the measurements above it are not.

1. The architecture you're actually running

When this dashboard tests a local model, it sends an HTTP request straight to LM Studio's REST API on localhost:1234. Hermes Desktop is never in that path — it doesn't load, eject, or route anything for these runs. That's the whole point: a clean-room measurement means nothing but the model itself sits between the prompt and the answer.

When this dashboard tests a cloud model, it calls the provider (Nous, OpenRouter, Anthropic, etc.) directly with its own API key — again, not through Hermes.

Hermes Desktop is a separate, parallel consumer of the same LM Studio server. Your chats with Hermes and this dashboard's benchmark runs can both talk to LM Studio, but Hermes' chat loop and this dashboard's test loop never touch each other.

2. Configure Hermes' main model (your daily driver)

In Hermes Agent, your main model is what answers you in chat. Two equivalent ways to set it:

  • Desktop app: Settings → Model Settings → MAIN MODEL row → CHANGE. Shown as provider · model, e.g. openrouter · anthropic/claude-sonnet-5.
  • In any chat session: the /model slash command opens the same picker.

Pick cloud for daily driving if you want reasoning, vision, and tool-use all in one place — a real example most people recognize: openrouter · anthropic/claude-sonnet-5. The switch is live and persists to config.yaml automatically.

3. Configure auxiliary tasks — the part people skip

Hermes routes 11 background jobs (vision, web extract, context compression, skill search, command-approval scoring, MCP tool routing, session titles, and several Kanban/curator jobs) to auxiliary models that are separate from your main chat model. Every task defaults to auto — meaning it silently rides your main model, at your main model's price.

Best place to configure this: Desktop app → Settings → Model Settings → AUXILIARY TASKS row → CONFIGURE. That opens the full 11-slot panel where each row shows its current assignment (auto (use main model) or provider · model) with its own Change button — no CLI needed.

Real, typical examples — the kind of pairing most Hermes users actually run:

  • Vision → a cloud example: nous · stepfun/step-3.7-flash:free (fast, free-tier, has vision) — or a local example: lmstudio · qwen2.5-vl-7b-instruct if it's already loaded/downloaded in your LM Studio. That local key won't be findable in anyone else's LM Studio for a long time — it's only there once you've downloaded that exact model — which is exactly why the picker (next section) matters.
  • MCP tool routing → nous · stepfun/step-3.7-flash:free — cheap/free is fine here, it's just routing tool calls, not reasoning hard.
  • Approval (Smart auto-approve) — this is the actual "security slot." When approvals.mode: smart is set, THIS auxiliary task is the model that decides whether to auto-approve a risky-looking shell command instead of prompting you. Point it at something fast and cheap (haiku, flash, gpt-5-mini) — an expensive reasoning model here is wasted spend, not extra safety.

CLI equivalent, if you prefer terminal: hermes config set auxiliary.<task>.provider lmstudio then hermes config set auxiliary.<task>.model <model-key>. Task names match the desktop panel's internal keys: vision, web_extract, compression, skills_hub, approval, mcp, title_generation, triage_specifier, kanban_decomposer, profile_describer, curator.

One-time setup, not a flip-flop — set it once, verify, move on.

3b. Does the picker auto-find new local models, or do I have to hit refresh?

For LM Studio specifically: fully automatic, every time you open the picker. Verified straight from Hermes' source (hermes_cli/models.py): the LM Studio path calls its /api/v1/models endpoint live, with no disk cache in front of it — unlike most cloud providers, whose curated model lists ARE disk-cached for up to an hour. So the moment you download and load a new model in LM Studio, it shows up in Hermes' picker the very next time you open it. No action needed.

The --refresh flag (/model --refresh, or the desktop picker's refresh control) exists for the other providers — it busts that hour-long disk cache so a newly-released cloud model (say, a new free OpenRouter model) shows up immediately instead of waiting out the cache window. It's harmless to hit on LM Studio too — just a no-op there, since there was never a cache to clear.

4. Why local models feel different — and how to use this dashboard to fix that

The honest gap between local and cloud isn't intelligence alone — it's speed. This dashboard measures average response latency per axis, in milliseconds, right next to the verdict. Check the 🏆 Loot tab for a leaderboard and a recommended squad — the same "one model per slot" idea as the auxiliary tasks panel above, but backed by your own measured evidence instead of guessing.

Use the Prompt Length Checker on the Benchmark page before sending anything heavy to a local model — it gives you an instant, zero-cost estimate of whether your prompt fits that model's context window. LM Studio's REST API has no live tokenizer endpoint, so the default check is a calibrated heuristic (~3.5 chars/token + 20% safety margin); a "Verify exact count" button is there when you want the real number, at the cost of one small real inference call.

MCP: Sampling Relay
An MCP tool server is asking you to answer a short factual question on its behalf, as part of a sampling request. Answer with ONLY the number, no explanation, no punctuation. What is 17 multiplied by 23?
Logic: Modus Ponens
Premise 1: If it is raining, then the ground is wet. Premise 2: It is raining. Conclusion: The ground is wet. Is this argument valid? Answer VALID or INVALID.
Logic: Affirming the Consequent (Fallacy)
Premise 1: If the database is corrupted, then queries fail. Premise 2: Queries are failing. Conclusion: The database is corrupted. Is this argument valid? Answer VALID or INVALID.
Logic: Syllogism Barbara
Premise 1: All security vulnerabilities are bugs. Premise 2: All buffer overflows are security vulnerabilities. Conclusion: All buffer overflows are bugs. Is this syllogism valid? Answer VALID or INVALID.
Logic: Boolean De Morgan
Expression: NOT (A AND B) is logically equivalent to (NOT A) OR (NOT B). Is this equivalence correct? Answer TRUE or FALSE.
Logic: Satisfiability
Formula: (A OR B) AND (NOT A OR C) AND (NOT B OR NOT C) Is this formula satisfiable? Answer SAT or UNSAT.
2. Test It — live inference against your chosen model
🎯 Why This Matters — the blueberry principle

You sent someone to the store for blueberries. They came back with tiny, mushy ones. Not because they're stupid — because you didn't say "large, plump, fresh blueberries."

Same with models. "Summarize this" gets you a paragraph. "Summarize this in 3 bullet points, each under 15 words, focusing on action items" gets you what you actually need.

The model doesn't know what you mean. It only knows what you say. This page teaches you to say exactly what you mean — and verify it works before you trust it in production.

🎯 The Genie Test

There's an 80s/90s movie where a guy wishes to be rich. Genie snaps — mansion, cars, women. He thinks he won. Then he gets shot because the genie made him a drug lord to pay for it. Be precise about what you want. The genie executes literally.
